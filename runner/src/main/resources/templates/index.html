Key_Shared — Routage garanti par clé métier

exclusif Pulsar
✅ Pulsar — Key_Shared natif ❌ Artemis — pas d'équivalent

Avec SubscriptionType.Key_Shared, Pulsar garantit que tous les messages portant la même clé (ex. un contractId) sont toujours livrés au même consumer, même avec N consumers parallèles partageant la subscription. Chaque clé est assignée à un consumer via un anneau de hachage géré par le broker. Impossible à garantir nativement avec Artemis en mode concurrent.

Dead Letter Topic — protection contre les messages empoisonnés

exclusif Pulsar
✅ Pulsar — DLT natif par subscription ❌ Artemis — configuration broker manuelle

Quand un consumer appelle negativeAcknowledge() sur un message répétitivement (maxRedeliverCount=2), Pulsar le reroute automatiquement vers un Dead Letter Topic dédié — sans perte de données. Ici, 1 message sur 3 simule une erreur de traitement métier. Les messages sains sont acquittés normalement ; les messages empoisonnés atterrissent dans le DLT où un consumer dédié les récupère pour audit ou re-traitement manuel.

Fan-out multi-subscription — broadcast indépendant

exclusif Pulsar
✅ Pulsar — curseurs indépendants par subscription ⚠ Artemis — topics séparés requis

Deux subscriptions de noms différents sur le même topic reçoivent chacune l'intégralité du flux, indépendamment l'une de l'autre. audit-trail et analytics-engine partagent le même topic source sans coordination ni duplication de données. Avec Artemis en mode queue, il faudrait deux queues distinctes et un routage applicatif.

Message Replay — Relecture après acquittement

exclusif Pulsar
✅ Pulsar — seek(earliest) natif ❌ Artemis — messages détruits après ack

Pulsar conserve les messages même après acquittement grâce à la rétention configurable sur le namespace (public/demo configuré ici à 60 min / 100 MB). Un consumer peut appeler consumer.seek(MessageId.earliest) pour rejouer l'historique depuis le début — utile pour la reprise sur incident, la migration de données ou le recalcul d'agrégats.
