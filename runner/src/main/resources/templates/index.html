Key_Shared — Routage garanti par clé métier i Key_Shared (Pulsar) : Associe chaque clé de message (ex: CTR-A, CTR-B) à un consommateur unique via un anneau de hachage logique géré par le broker. Si un consommateur tombe en panne, le broker redirige ses clés de manière déterministe sans affecter les autres flux ni perturber l'ordre.

exclusif Pulsar
Pulsar — Key_Shared natif Artemis — pas d'équivalent en queue concurrentielle

Avec le mode de souscription Key_Shared, Apache Pulsar garantit que tous les messages portant la même clé métier (par exemple, un identifiant de contrat d'assurance) sont toujours remis au même consommateur, permettant un traitement ordonné en parallèle. Artemis ne dispose pas de routage par clé dynamique natif en mode concurrent (les Message Groups d'Artemis requièrent un partitionnement statique complexe).

Dead Letter Topic — messages empoisonnés i Dead Letter Topic (DLT) : Permet de décharger et isoler les messages posant problème (exceptions applicatives répétées) afin de ne pas bloquer les messages sains suivants. Le broker gère nativement le DLT par subscription.

exclusif Pulsar
Pulsar — DLT natif par subscription Artemis — configuration broker manuelle globale

Si un traitement échoue et renvoie un acquittement négatif negativeAcknowledge() plusieurs fois de suite (limite maxRedeliverCount=2), Pulsar déplace automatiquement le message empoisonné vers un Dead Letter Topic (DLT) dédié. Le consommateur principal peut continuer son travail sans être bloqué. Fan-out multi-subscription — broadcast indépendant i Fan-out multi-subscription : Pulsar permet de brancher plusieurs abonnements (subscriptions) autonomes sur un seul topic. Chaque abonnement possède son propre pointeur (curseur) et lit l'intégralité du flux de manière totalement isolée.

exclusif Pulsar
Pulsar — curseurs indépendants par subscription Artemis — queues distinctes et routage requis

Deux abonnements de noms distincts branchés sur le même topic physique reçoivent chacun l'intégralité du flux de messages de façon 100% indépendante. Ici, les microservices factices audit-trail et analytics-engine lisent en parallèle sans coordination ni duplication de données sur le broker. Sur Artemis, cela nécessiterait de déclarer des files d'attente distinctes et de configurer du routage.

Message Replay — Relecture après acquittement i Message Replay : Grâce au stockage structuré d'Apache BookKeeper, Pulsar conserve l'historique physique des messages même s'ils sont acquittés, selon des règles de rétention. Les consommateurs peuvent repositionner leur curseur dans le passé.

exclusif Pulsar
Pulsar — seek(earliest) natif Artemis — messages détruits immédiatement après ack

Pulsar conserve les messages dans un journal ordonné et persistant après acquittement grâce à la rétention configurable. Un consommateur peut appeler consumer.seek(MessageId.earliest) pour relire le flux depuis le début du journal. C'est idéal pour recalculer des données (Event Sourcing), corriger un bug métier historique, ou rattraper un crash.
